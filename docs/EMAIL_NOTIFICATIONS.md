## Email Notifications via Firebase Extension

This document describes how the Pharmacy Logistics system sends **email alerts** using the official Firebase email extension and how it ties into existing notification preferences.

---

### 1. Overview

- **Goal**: Send email alerts to staff when there are **expired medicines**, **expiring soon medicines**, or **low stock** items.
- **Trigger Source**: Existing `alerts` collection in Firestore (created automatically by inventory logic).
- **Delivery Mechanism**: The official Firebase **Trigger Email** extension (listens to documents created in a Firestore collection and sends emails).
- **Client Integration**: The web app writes documents into the `mail` collection whenever an alert is created *and* the current user has enabled email notifications with **immediate** frequency.

This approach requires **no custom Cloud Functions code** in this repository; all email sending is handled by the Firebase extension.

### 3. Install the Firebase Email Extension

Use the official **Trigger Email** extension (Firestore â†’ Email):

1. Open the Firebase Console for your project.
2. Go to **Extensions** â†’ **Discover extensions**.
3. Search for **"Trigger Email"** (Firestore-triggered email extension by Firebase).
4. Click **Install** and select your project: `ehafo-pharmacy-logistics-3ca06`.
5. When configuring the extension, use the following key settings:
   - **Collection path**: `mail`
   - **Sender email address**: The email you want to appear as "from" (e.g. `no-reply@your-clinic.com`).
   - **Email service / region**: As recommended or suitable for your project.
6. Complete the installation.

> After installation, the extension will watch the `mail` collection in Firestore. Any new document with the correct shape will result in an email being sent.

### 4. Data Model

#### 4.1 Notification Preferences (existing)

User-level notification preferences are stored in:

- **Collection**: `notificationPreferences`
- **Document ID**: user UID (`auth.currentUser.uid`)
- **Schema** (simplified):
  - `emailEnabled` (boolean) â€“ whether the user wants email alerts.
  - `emailFrequency` (string) â€“ one of `immediate`, `daily`, `weekly`.
  - `types` (object):
    - `expired` (boolean)
    - `expiry_soon` (boolean)
    - `low_stock` (boolean)
  - `lastEmailSent` (timestamp or null) â€“ reserved for future scheduling logic.

These preferences are managed by the `Notification Settings` page (`notifications-settings.html`) and saved using `getNotificationPreferences` / `saveNotificationPreferences` in `notifications.js`.

#### 4.2 Alerts (existing)

Alerts are generated by inventory logic in `inventory.js` and stored in:

- **Collection**: `alerts`
- **Key fields** (simplified):
  - `medicineId`, `medicineName`
  - `type` â€“ `expired`, `expiry_soon`, `low_stock`
  - `dueOn` â€“ expiry date for `expired` / `expiry_soon`
  - `severity` â€“ `high`, `medium`, `low`
  - `resolved` â€“ whether the alert is resolved.

Whenever a new alert is created, the app calls `triggerAlertNotification(alert)` in `alerts.js`.

#### 4.3 Mail Queue (for the extension)

The email extension listens to documents in:

- **Collection**: `mail`
- **Document shape (used by this app)**:
  - `to` (string): recipient email (current user).
  - `message` (object):
    - `subject` (string)
    - `text` (string)
    - `html` (string)
  - **Metadata (optional but useful)**:
    - `alertType` (string): `expired` | `expiry_soon` | `low_stock`
    - `medicineId` (string or null)
    - `medicineName` (string or null)
    - `queuedByUid` (string): UID of the user that queued the email.
    - `createdAt` (timestamp): `firebase.firestore.FieldValue.serverTimestamp()`

The extension uses `to` and `message.*` to construct and send the email. The metadata fields are purely for logging and troubleshooting.

### 5. Frontend Integration

The main integration lives in `frontend/js/alerts.js`.

#### 5.1 High-level flow

1. Inventory logic in `inventory.js` detects a problem (expired, expiring soon, or low stock).
2. It calls `createOrUpdateAlert(...)`, which creates a document in the `alerts` collection.
3. After creating an alert, `triggerAlertNotification(alert)` is called.
4. `triggerAlertNotification`:
   - Updates the in-app notification center and browser notification (existing behavior).
   - Calls `queueEmailNotificationForAlert(alert)` to enqueue an email (new behavior).
5. `queueEmailNotificationForAlert`:
   - Reads the current user and their `notificationPreferences` from Firestore.
   - If email notifications are enabled and the alert type is allowed, and the frequency is `immediate`, it writes a document to the `mail` collection.
6. The Firebase **Trigger Email** extension picks up the document in `mail` and sends an email.

#### 5.2 `queueEmailNotificationForAlert` behavior

In `alerts.js`:

- Checks that Firestore (`db`) and `firebase` are available.
- Retrieves the current authenticated user via `getCurrentUser()` from `auth.js`.
- Retrieves the userâ€™s notification preferences from `notificationPreferences/{uid}`.
- Enforces:
  - `emailEnabled === true`
  - Alert type is not disabled under `prefs.types`.
  - `emailFrequency === 'immediate'` (for now, only immediate emails are implemented).
- Builds a clear, human-readable **subject** and **body** (text + HTML) based on the alert type and expiry date.
- Adds a document to the `mail` collection with the expected fields for the extension.

If any step fails, the error is logged to the console but does **not** affect in-app or browser notifications.

### 6. Respecting User Settings

The Notification Settings page (`notifications-settings.html`) exposes the following controls:

- **Enable Email Notifications** (checkbox â†’ `emailEnabled`).
- **Notification Frequency**:
  - `immediate`
  - `daily`
  - `weekly`
- **Alert Types** (checkboxes):
  - Expired medicines
  - Expiring soon
  - Low stock

Current implementation details:

- **Immediate**: When `emailFrequency` is `immediate`, emails are sent for each qualifying alert according to the alert type switches.
- **Daily / Weekly**: These modes are **stored** in Firestore but are **not yet implemented** in the backend. No emails are sent for those frequencies until a scheduled backend job is added.

Planned future enhancement:

- Add a Cloud Function (or other scheduler) that runs daily/weekly, aggregates alerts per user, and writes summary documents to the `mail` collection based on `lastEmailSent` and `emailFrequency`.

### 7. How to Verify the Integration

1. **Install and configure** the Trigger Email extension (see section 3).
2. **Enable email notifications** for a test account:
   - Log in to the app.
   - Go to **Notification Settings**.
   - Enable **Email Notifications**.
   - Set frequency to **Immediate**.
   - Ensure relevant alert type checkboxes are ticked.
3. **Trigger an alert**:
   - Create or update medicine data so that it is **expired**, **expiring soon**, or **low stock**.
4. **Observe behavior**:
   - In the Firebase Console, open Firestore and watch the `mail` collection.
   - A new document should appear corresponding to your alert.
   - Check the email inbox of the user account you used to sign in.
   - You should receive an email with the subject/body describing the alert.

If you see documents in `mail` but no emails, the issue is likely with **extension configuration** or your email service credentials.

### 8. Security Considerations

- Only authenticated users can trigger the alert flow, and thus enqueue email documents.
- Documents in the `mail` collection are written from the client.
  - For production, you may want to restrict direct client writes to `mail` and instead use a Cloud Function to mediate writes.
- Update your Firestore Security Rules to ensure that only trusted paths/fields can be written and that users cannot abuse the `mail` collection to send arbitrary emails.

At minimum, ensure that:

- Only authenticated users with appropriate roles can write documents that lead to emails being sent.
- User-supplied strings are sanitized or constrained as needed.

- The frontend writes `mail` documents; the extension takes care of actually sending emails.
- Immediate per-alert emails are implemented today; daily/weekly summaries are planned as a future backend enhancement.
