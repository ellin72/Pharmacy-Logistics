## Email Notifications via SMTP (Cloud Functions)

This document describes how the Pharmacy Logistics system sends **email alerts** using **SMTP** via a Firebase Cloud Function, and how it ties into existing notification preferences.

---

### 1. Overview

- **Goal**: Send email alerts to staff when there are **expired medicines**, **expiring soon medicines**, or **low stock** items.
- **Trigger Source**: Existing `alerts` collection in Firestore (created automatically by inventory logic).
- **Delivery Mechanism**: A Firebase **Cloud Function** that listens to documents created in the `mail` collection and sends emails via **SMTP** using Nodemailer.
- **Client Integration**: The web app writes documents into the `mail` collection whenever an alert is created *and* the current user has enabled email notifications with **immediate** frequency.

Compared to the Firebase email extension, this approach gives you **full control** over SMTP credentials and sending behavior in code.

---

### 2. Prerequisites

- Firebase project: `ehafo-pharmacy-logistics-3ca06` (from `frontend/js/config.js`).
- Firestore and Authentication already enabled (they are used throughout the app).
- You must have **Owner** or **Editor** access to the Firebase project and Firebase CLI installed.
- An SMTP provider (e.g. your clinic's mail server, SendGrid, Mailgun, Gmail SMTP if compliant with your policies).

### 3. Cloud Functions Setup (SMTP)

The repository now includes a `functions` directory with a Cloud Function that sends emails via SMTP.

#### 3.1 Install dependencies

From the project root (`Pharmacy-Logistics`):

```bash
cd functions
npm install
```

This installs:

- `firebase-functions` – Cloud Functions SDK
- `firebase-admin` – Admin access to Firestore
- `nodemailer` – SMTP email library

#### 3.2 Configure SMTP credentials

Configure SMTP using Firebase Functions config (never hardcode secrets in code):

```bash
firebase functions:config:set \
  smtp.host="YOUR_SMTP_HOST" \
  smtp.port="587" \
  smtp.secure="false" \
  smtp.user="YOUR_SMTP_USERNAME" \
  smtp.pass="YOUR_SMTP_PASSWORD" \
  smtp.from="Your Clinic <no-reply@your-clinic.com>"
```

Notes:

- `smtp.secure` should be `true` if you use port `465`, otherwise `false` for `587`/`25` with STARTTLS.
- `smtp.from` sets the sender that appears in inboxes.

You can verify the config at any time:

```bash
firebase functions:config:get smtp
```

#### 3.3 Deploy the function

From the project root:

```bash
firebase deploy --only functions
```

This deploys the `sendMailOnCreate` function defined in `functions/index.js`.

### 4. Data Model

#### 4.1 Notification Preferences (existing)

User-level notification preferences are stored in:

- **Collection**: `notificationPreferences`
- **Document ID**: user UID (`auth.currentUser.uid`)
- **Schema** (simplified):
  - `emailEnabled` (boolean) – whether the user wants email alerts.
  - `emailFrequency` (string) – one of `immediate`, `daily`, `weekly`.
  - `types` (object):
    - `expired` (boolean)
    - `expiry_soon` (boolean)
    - `low_stock` (boolean)
  - `lastEmailSent` (timestamp or null) – reserved for future scheduling logic.

These preferences are managed by the `Notification Settings` page (`notifications-settings.html`) and saved using `getNotificationPreferences` / `saveNotificationPreferences` in `notifications.js`.

#### 4.2 Alerts (existing)

Alerts are generated by inventory logic in `inventory.js` and stored in:

- **Collection**: `alerts`
- **Key fields** (simplified):
  - `medicineId`, `medicineName`
  - `type` – `expired`, `expiry_soon`, `low_stock`
  - `dueOn` – expiry date for `expired` / `expiry_soon`
  - `severity` – `high`, `medium`, `low`
  - `resolved` – whether the alert is resolved.

Whenever a new alert is created, the app calls `triggerAlertNotification(alert)` in `alerts.js`.

#### 4.3 Mail Queue (for SMTP function)

The SMTP Cloud Function listens to documents in:

- **Collection**: `mail`
- **Document shape (used by this app)**:
  - `to` (string): recipient email (current user).
  - `message` (object):
    - `subject` (string)
    - `text` (string)
    - `html` (string)
  - **Metadata (optional but useful)**:
    - `alertType` (string): `expired` | `expiry_soon` | `low_stock`
    - `medicineId` (string or null)
    - `medicineName` (string or null)
    - `queuedByUid` (string): UID of the user that queued the email.
    - `createdAt` (timestamp): `firebase.firestore.FieldValue.serverTimestamp()`

The function uses `to` and `message.*` to construct and send the email via SMTP. The metadata fields are for logging and troubleshooting.

### 5. Cloud Function Behavior

In `functions/index.js`, the function `sendMailOnCreate` is defined as:

- **Trigger**: `onCreate` of documents in `mail/{mailId}` (Firestore collection `mail`).
- **Region**: `europe-west1` (you can change this if needed).
- **Steps**:
  1. Reads the newly created document.
  2. Validates SMTP configuration (`smtp.host`, `smtp.from`, etc.).
  3. Validates that `to`, `message.subject`, and `message.text` are present.
  4. Uses Nodemailer to send the email via the configured SMTP server.
  5. Writes back to the same document:
     - `sent: true/false`
     - `sentAt`: server timestamp
     - `smtpMessageId`: ID returned by the SMTP server (if available)
     - `error`: error message if sending fails

If SMTP is misconfigured or sending fails, errors are logged and written to the Firestore document for easy debugging.

### 6. Frontend Integration

The main frontend integration remains in `frontend/js/alerts.js`.

#### 6.1 High-level flow

1. Inventory logic in `inventory.js` detects a problem (expired, expiring soon, or low stock).
2. It calls `createOrUpdateAlert(...)`, which creates a document in the `alerts` collection.
3. After creating an alert, `triggerAlertNotification(alert)` is called.
4. `triggerAlertNotification`:
   - Updates the in-app notification center and browser notification (existing behavior).
   - Calls `queueEmailNotificationForAlert(alert)` to enqueue an email (this is already implemented in the frontend).
5. `queueEmailNotificationForAlert`:
   - Reads the current user and their `notificationPreferences` from Firestore.
   - If email notifications are enabled, the alert type is allowed, and the frequency is `immediate`, it writes a document to the `mail` collection.
6. The **SMTP Cloud Function** picks up the document in `mail` and sends an email via your SMTP server.

#### 6.2 Respecting User Settings

The Notification Settings page (`notifications-settings.html`) exposes the following controls:

- **Enable Email Notifications** (checkbox → `emailEnabled`).
- **Notification Frequency**:
  - `immediate`
  - `daily`
  - `weekly`
- **Alert Types** (checkboxes):
  - Expired medicines
  - Expiring soon
  - Low stock

Current implementation details:

- **Immediate**: When `emailFrequency` is `immediate`, emails are sent for each qualifying alert according to the alert type switches.
- **Daily / Weekly**: These modes are **stored** in Firestore but are **not yet implemented** in the backend. No scheduled summary emails are sent until a future scheduler is added.

### 7. How to Verify the SMTP Integration

1. **Configure SMTP** (section 3.2) and **deploy functions** (section 3.3).
2. **Enable email notifications** for a test account:
   - Log in to the app.
   - Go to **Notification Settings**.
   - Enable **Email Notifications**.
   - Set frequency to **Immediate**.
   - Ensure relevant alert type checkboxes are ticked.
3. **Trigger an alert**:
   - Create or update medicine data so that it is **expired**, **expiring soon**, or **low stock**.
4. **Observe behavior**:
   - In the Firebase Console, open Firestore and watch the `mail` collection.
   - A new document should appear corresponding to your alert.
   - After the function runs, that document should update with `sent: true` and `sentAt`.
   - Check the email inbox of the user account you used to sign in.
   - You should receive an email with the subject/body describing the alert.

If documents appear in `mail` but emails fail:

- Check the Firestore document for `error` fields.
- Check Cloud Functions logs in Firebase Console → Functions → Logs.
- Verify your SMTP credentials and that your SMTP provider accepts connections from the Cloud Functions environment.

---

### 8. Security Considerations

- SMTP credentials are stored **only** in Functions config (`functions.config().smtp`), not in the repository.
- The client only writes to the `mail` collection; it never sees SMTP secrets.
- Update your Firestore Security Rules to ensure that only authenticated users with appropriate roles can write to the `mail` collection, and that they cannot abuse it to send arbitrary spam emails.
- Consider adding simple server-side checks in the function (e.g. rate limiting by `queuedByUid`, validating email formats) for additional protection.

---

### 9. Summary

- The Pharmacy Logistics system now uses a Firebase Cloud Function and **SMTP** to send email alerts.
- The frontend continues to queue emails by writing to the `mail` collection, respecting per-user notification preferences.
- The Cloud Function reads `mail` documents, uses Nodemailer with your SMTP configuration, and writes back status fields for observability.
- Immediate per-alert emails are implemented; daily/weekly summary emails remain a possible future enhancement.
